# 1. Введение: Зачем нужен TLS?

Изначально интернет строился на абсолютном доверии. Базовый протокол веба, **HTTP**, передает данные в открытом виде (plaintext). В начале 90-х годов это никого не волновало — интернет использовали ученые для обмена статьями. 

Но к 1994 году стало понятно, что за интернетом коммерческое будущее. Люди хотели покупать вещи онлайн, а значит — передавать номера кредитных карт. Передавать кредитку в открытом виде — всё равно что писать её номер на почтовой открытке: любой узел по пути (ваш провайдер или хакер) может её прочитать или изменить (это называется **атака Man-in-the-Middle**).

Чтобы спасти зарождающуюся электронную коммерцию, за дело взялась компания **Netscape Communications**.

## История версий

| Год | Версия | Описание |
|-----|--------|---------|
| 1994 | **SSL 1.0** | Инженеры Netscape создали протокол Secure Sockets Layer. Первая версия была настолько уязвимой, что не вышла в релиз и осталась внутренним черновиком. |
| 1995 | **SSL 2.0** | Первая публичная версия с браузером Netscape Navigator. Позволила запустить первые безопасные платежи, но содержала серьезные уязвимости. |
| 1996 | **SSL 3.0** | Netscape привлекла сторонних криптографов (в частности, Пола Кохера) и полностью переписала протокол. Получился удачным и стал де-факто мировым стандартом. |
| 1999 | **TLS 1.0** | Разработку передали IETF (Инженерному совету интернета). Протокол переименовали в TLS (Transport Layer Security). Технически это было минорное обновление SSL 3.0 (версия 3.1). |
| 2006 | **TLS 1.1** | Эволюция протокола |
| 2008 | **TLS 1.2** | Стал настоящим «рабочим конем» интернета и массово используется до сих пор |
| 2018 | **TLS 1.3** | Современный стандарт с улучшениями безопасности и производительности |

### Текущий статус

- **Устаревшие**: Все версии SSL и TLS до 1.1 включительно признаны устаревшими и не рекомендуются к использованию
- **Термин SSL-сертификат** остался в обиходе несмотря на переименование на TLS

### TLS в сетевой модели OSI

В сетевой модели OSI протокол TLS является своеобразной прослойкой:
- Работает **поверх транспортного уровня** (TCP)
- Инкапсулирует в себя протоколы **прикладного уровня**
- HTTP, обернутый в TLS-шифрование, становится безопасным **HTTPS**

# 2. Базовая механика

TLS держится на **трех китах**:

## Три основных принципа

### 1. **Шифрование**
- Гарантия того, что никто посторонний данные не прочитает
- Использует симметричное шифрование после хэндшейка

### 2. **Аутентификация**
- Гарантия того, что мы общаемся именно с сервером нашего банка, а не с мошенником
- Механика SSL сертификатов и их подписей

### 3. **Целостность**
- Гарантия того, что данные не были изменены по пути (к сумме перевода не дописали лишний ноль)
- **До TLS 1.2**: подпись/проверка подлинности данных с помощью HMAC
- **TLS 1.3**: эту роль берут на себя алгоритмы симметричного шифрования

## Симметричное vs Асимметричное шифрование

### Симметричное шифрование

```
Клиент и Сервер имеют общий секретный ключ
    ↓
Оба шифруют и расшифровывают данные этим ключом
    ↓
Быстро и эффективно для больших объемов
```

**Преимущества:**
- Очень быстро
- Не нагружает CPU (за счет оптимизаций самого CPU)
- Отлично подходит для передачи большого количества данных

**Огромная проблема:**
- Как безопасно передать этот ключ по незащищенной сети, чтобы его никто не перехватил?

### Асимметричное шифрование (криптография)

```
Есть два ключа: публичный и приватный
    ↓
Публичный ключ: можно передавать по сети
Приватный ключ: нельзя, это секрет
    ↓
Кто угодно может зашифровать сообщение публичным ключом
    ↓
Расшифровать может только владелец приватного ключа
```

**Проблема:**
- Требует сложных математических вычислений
- Работает очень медленно

## Как TLS объединяет оба подхода

TLS берет **лучшее от обеих идей**:

1. **В начале сеанса** — медленное асимметричное шифрование используется только для handshake
2. **Договариваются** о секретном сессионном ключе
3. **Для передачи данных** — быстрое симметричное шифрование

## Обмен ключами: RSA

### Процесс:

1. **Client Hello** — Клиент (браузер) и сервер обмениваются приветствиями (сообщения `ClientHello` и `ServerHello`), передают друг другу случайные числа

2. **Certificate** — Сервер отправляет клиенту свой SSL-сертификат с открытым ключом сервера

3. **Pre-Master Secret** — Клиент генерирует случайную последовательность байтов (Pre-Master Secret)

4. **Encrypt & Send** — Клиент шифрует этот секрет открытым ключом сервера и отправляет по сети

5. **Only server can decrypt** — Расшифровать может только сервер (нужен его закрытый ключ, который никогда не покидает сервер)

6. **Shared Secret** — Сервер расшифровывает. Теперь у обеих сторон есть общий Pre-Master Secret

![Пример](https://sun9-33.vkuserphoto.ru/s/v1/ig2/tWFlzDJB6PsWs6vTSiLvS_OAY7uYpLsZ_Gn-11qkjoPf9Xk1v-lQVOoRJ-RYhbDv-FDCQNgMx37hgeevxzcpJp5j.jpg?quality=95&as=32x23,48x35,72x52,108x78,160x115,240x173,360x259,480x346,540x389,640x461,709x511&from=bu&u=Pwjky7NHtQmxtJOiI44xrt4KqBK61f5RPbhiNqvAQls&cs=709x0)

## Обмен ключами: Diffie–Hellman

**Ключевое отличие:** Секрет **Не передаётся** по сети вообще. Стороны независимо вычисляют одинаковый shared secret.

### Публичные параметры:
- **p** — большое простое число
- **g** — генератор

### Шаги:

```
Клиент:
  1. Генерирует секрет: a = random()
  2. Вычисляет: A = g^a mod p
  3. Отправляет на сервер: A

Сервер:
  1. Генерирует секрет: b = random()
  2. Вычисляет: B = g^b mod p
  3. Отправляет на клиент: B

Вычисление общего секрета:
  Клиент: s = B^a mod p
  Сервер: s = A^b mod p
  Результат: s = g^(ab) mod p ✓ (одинаково!)
```

![Пример](https://sun9-6.vkuserphoto.ru/s/v1/ig2/9HwAw-9JCB-ys8V7GJGhZKMRsI0qmlMenUEmyaI4GKgyz8j4isnOO9G6K9hIVEpzHvknVr9KEpX5TKaftwWdFNKH.jpg?quality=95&as=32x19,48x29,72x43,108x64,160x95,240x143,360x214,480x285,540x321,640x380,720x428,1080x642,1280x761,1302x774&from=bu&u=KPcPg-8ouwRdMN8UOY16ApcQ_uwipS8M4EYKMSdOa94&cs=1302x0)

### Итог:
- Из Pre-Master Secret генерируются ключи симметричного шифрования

# 3. Аутентификация: Сертификаты и цепочка доверия

Мы установили надежно зашифрованный канал связи. Но **с кем**? Какая разница, насколько стойкое у нас шифрование, если на другом конце провода сидит злоумышленник, а не наш банк?

## Цифровые сертификаты

Чтобы браузер понимал, что `google.com` — это реально Google, используются **цифровые сертификаты**.

### Содержимое сертификата:
- **Имя домена** (кому выдан)
- **Публичный ключ сервера**
- **Срок действия**
- **Цифровая подпись** организации, которая этот сертификат выдала (Certificate Authority, CA)

## Chain of Trust — Цепочка доверия

Но почему наш браузер должен верить этой организации? Здесь работает механизм **Chain of Trust** (Цепочка доверия).

### Основная идея:

```
1. CA подписывает сертификат своим private key
2. Клиент проверяет подпись public key'ем CA
```

### Процесс подписи (CA):

```
hash = HASH(certificate_data)
signature = SIGN(hash, CA_private_key)
```

### Процесс проверки (Клиент):

```
hash1 = HASH(certificate_data)
hash2 = VERIFY(signature, CA_public_key)

if hash1 == hash2 → ✓ Всё отлично!
```

## Цепочка сертификатов

Когда вы заходите на сайт, сервер присылает:
1. **Свой сертификат**
2. **Один или несколько промежуточных сертификатов** (Intermediate)

### Как браузер проверяет цепочку:

```
1. Берет сертификат сайта
   ↓
2. Смотрит, кто его подписал
   ↓
3. Идет к промежуточному сертификату
   ↓
4. Смотрит, кто подписал его
   ↓
5. Идет по цепочке вверх...
   ↓
Пока не дойдет до Root CA
(корневого центра сертификации)
```

### Root CA — корневые центры

- Публичные ключи этих корневых центров **"вшиты"** в вашу операционную систему и браузер
- Microsoft, Apple, Mozilla обновляют эти списки
- Если цепочка сошлась к доверенному корневому сертификату — проверка пройдена
- В браузере загорается тот самый заветный **замочек**

# 4. Проверка целостности

## Почему нужна целостность?

До этого этапа мы решили две задачи:
1. **Как безопасно обменяться ключом** — Диффи-Хеллман
2. **Как убедиться, что сервер настоящий** — сертификаты

Теперь представим: у нас есть общий секретный ключ, мы начинаем шифровать им наши данные (например, сообщение "Перевести Пете 100 рублей") и отправляем по сети.

### Проблема:

- Само по себе шифрование обеспечивает только **конфиденциальность** (хакер не прочитает текст)
- **Но базовое шифрование НЕ защищает от модификации**
- Хакер, перехвативший зашифрованный пакет, может наугад изменить в нём пару битов (***Bit-flipping attack***)
- На сервере сообщение расшифруется с ошибкой, но если повезет хакеру, оно может превратиться в: "Перевести Пете **900** рублей"

### Решение:

Нам нужна **Integrity** (Целостность). Сервер должен точно знать, что данные не были повреждены или изменены в пути.

## HMAC (Раздельное шифрование и аутентификация)

В старых версиях TLS (до TLS 1.2 включительно) шифрование и проверка целостности были **двумя разными независимыми операциями**.

Для проверки целостности использовался:
- **MAC** (Message Authentication Code)
- **HMAC** (Hash-based MAC) — продвинутая версия

### 4-шаговый процесс HMAC:

#### Шаг 1: Генерация тега

```
Браузер берет исходный текст: "Перевести 100..."
    ↓
Добавляет специальный ключ для HMAC (созданный на этапе handshake)
    ↓
Пропускает через хэш-функцию (например, SHA-256)
    ↓
Результат: MAC-тег (короткая уникальная строка — "печать целостности")
```

#### Шаг 2: Объединение

```
MAC-тег приклеивается в конец исходного сообщения
    ↓
[Сообщение] + [MAC-тег]
```

#### Шаг 3: Шифрование

```
Всё это вместе (сообщение + MAC-тег) шифруется симметричным ключом
    ↓
Алгоритм: AES в режиме CBC
    ↓
Отправляется на сервер
```

#### Шаг 4: Проверка на сервере

```
Сервер получает пакет
    ↓
Расшифровывает его
    ↓
Отделяет MAC-тег
    ↓
Берет текст, вычисляет для него HMAC
    ↓
Сравнивает с полученным тегом
    ↓
Совпало → ✓ данные целы
```

## Почему HMAC устарел?

### Проблемы:

**Слишком сложно реализовать безопасно**

Выполнение шифрования и хэширования как двух отдельных шагов оказалось "минным полем". Программистам приходилось решать:
- **Encrypt-then-MAC** (сначала шифровать, потом хэшировать)
- **MAC-then-Encrypt** (сначала хэшировать, потом шифровать)

**Исторически TLS выбрали второй путь**, что привело к серии уязвимостей:
- **Padding Oracle атаки**
- **Lucky Thirteen (Lucky13)** — знаменитая уязвимость

## AEAD (Всё-в-одном)

Из-за проблем с раздельным подходом криптографы придумали **AEAD** (Authenticated Encryption with Associated Data — Аутентифицированное шифрование с присоединенными данными).

### AEAD в TLS 1.3:

- **Обязателен** только AEAD
- Запрещены все старые методы (HMAC+AES CBC, MD5, SHA-1, RC4)
- Разрешены: **AES-GCM** и **ChaCha20-Poly1305**

### Суть AEAD

Шифрование и проверка целостности выполняются **одновременно**, за **одну математическую операцию**.

### AE — Authenticated Encryption:

```
Алгоритм берет:
  ✓ Открытый текст
  ✓ Симметричный ключ
    ↓
Пропуская их через себя, выдает два результата:
  1. Зашифрованный текст (Ciphertext)
  2. Аутентификационный тег (Auth Tag) — "печать целостности"
```

### AD — Associated Data (Присоединенные данные):

В любом сетевом пакете TLS есть **служебная информация**:
- Версия TLS
- Длина пакета
- Заголовки

**Её нельзя шифровать** (нужна для правильной доставки)
**Но её нужно защитить от подделки**

**В AEAD:** Эту открытую информацию можно "скормить" алгоритму как **Присоединенные данные** (AD):
- Алгоритм **не будет** её шифровать
- Но **учтёт** при создании Auth Tag

### Что происходит при получении пакета AEAD:

```
Сервер получает:
  ✓ Зашифрованный текст
  ✓ Открытые заголовки
  ✓ Auth Tag
    ↓
Алгоритм дешифрования начинает работу
    ↓
ЕСЛИ хакер изменил хотя бы один бит:
  • В зашифрованном тексте ИЛИ
  • В открытом заголовке (например, подменил длину пакета)
    ↓
Алгоритм при проверке Auth Tag мгновенно выдаст ошибку
    ↓
Отличие от старого подхода:
  Алгоритм AEAD сразу отбросит испорченный пакет
  И даже не будет пытаться выдать расшифрованный "мусор"
    ↓
Соединение разрывается: ошибка bad_record_mac
```

# 5. TLS Handshake (Рукопожатие) на примере TLS 1.2

Давайте заглянем под капот и посмотрим, как именно клиент и сервер договариваются о связи. Этот процесс называется **TLS Handshake** (рукопожатие). 

Рассмотрим его на примере всё ещё популярного протокола **TLS версии 1.2**.

## Этапы рукопожатия (4 шага)

### Шаг 1: Client Hello

```
Ваш браузер стучится к серверу и говорит:
  "Привет! Я хочу общаться безопасно.
   
   Вот максимальная версия TLS, которую я знаю
   И вот список шифров (Cipher Suites), которые я поддерживаю
   
   Выбирай"
```

**Браузер отправляет:**
- Максимальная версия TLS
- Список поддерживаемых cipher suites (комбинаций алгоритмов)

### Шаг 2: Server Hello + Certificate

```
Сервер отвечает:
  "Привет! Давай использовать вот этот конкретный шифр.
   
   А вот мой сертификат с публичным ключом —
   проверяй, что я это я"
```

**Сервер отправляет:**
- Выбранный шифр (из предложенных браузером)
- Свой SSL-сертификат с публичным ключом

### Шаг 3: Key Exchange

```
Клиент проверяет сертификат
(по цепочке доверия, о которой мы говорили)
    ↓
Затем стороны используют хитрую математику
(обычно алгоритм Диффи-Хеллмана)
    ↓
Генерируют общий секретный сессионный ключ
    ↓
Даже если кто-то перехватывал весь их диалог,
он не сможет собрать этот ключ воедино
```

**Математика:**
- Используется асимметричное шифрование (RSA или ECDHE)
- Генерируется симметричный сессионный ключ
- Ключ вычисляется независимо обеими сторонами

### Шаг 4: Finished

```
Стороны обмениваются сообщениями
    ↓
Подтверждают, что рукопожатие прошло успешно
    ↓
С этого момента всё будет зашифровано новым ключом
```

**Результат:**
- Оба конца подтвердили успешное рукопожатие
- Начинается передача зашифрованных данных

## Проблема TLS 1.2: Задержка

Как вы заметили, на этот процесс уходит **2 полных пути туда-обратно по сети** — или **2 RTT** (Round Trip Time).

```
RTT #1: Client Hello ────→
                    ←────── Server Hello + Certificate
                    
RTT #2: Key Exchange ────→
                    ←────── Finished
                    
════════════════════════════════════════════════════════════
После этого только NOW браузер может отправить:
           
        HTTP запрос (например, на картинку)
```

### Почему это проблема?

Прежде чем мы отправим **хотя бы первый байт полезных данных** (например, запрос на картинку):
- Пакеты должны **дважды облететь** половину Земного шара
- Для современного быстрого веба это **непозволительно долго**

# 6. TLS 1.3

Поэтому в 2018 году, после долгих лет разработки, был принят стандарт **TLS 1.3**. Что он нам дал?

## Ускорение

### 1 RTT вместо 2 RTT

В TLS 1.3 инженеры смогли сократить рукопожатие до **1 RTT**:

```
RTT #1: Client Hello (+ параметры) ────→
                                  ←────── Server Hello + Certificate + Finished
                                  
════════════════════════════════════════════════════════════
После ПЕРВОГО round-trip браузер может отправить:
           
        ЗАШИФРОВАННЫЙ HTTP запрос
```

### Процесс обмена ключами начинается в Client Hello

- Обмен ключами **начинается прямо в первом сообщении** `Client Hello`
- Серверу **больше не нужно ждать** второго круга
- Он **сразу присылает** свою часть ключа и сертификат

## 0-RTT: Возобновление сессии

Появилась фича **0-RTT** (Zero Round Trip Time):

```
Если вы вчера уже заходили на этот сайт
    ↓
Ваш браузер помнит прошлые параметры
    ↓
Может отправить ЗАШИФРОВАННЫЙ HTTP-запрос
    ↓
Прямо в САМОМ ПЕРВОМ пакете, вместе с приветствием!
    ↓
Задержка на криптографию = НОЛЬ
```

## Безопасность — Генеральная уборка

### Выкинули всё устаревшее

Из TLS 1.3 **полностью запрещена** поддержка старых, уязвимых алгоритмов:

Удалены:
- **MD5** (криптографическая хэш-функция)
- **SHA-1** (криптографическая хэш-функция)
- **RC4** (алгоритм шифрования)
- **Устаревший RSA** для обмена ключами

Оставлены только:
- Современная криптография
- Проверенные алгоритмы

### Perfect Forward Secrecy — Совершенная прямая секретность

Особенно важное улучшение!

#### Что происходило в старых версиях:

```
Хакер:
  1. Годами записывает твой зашифрованный трафик
  2. Ждет 10 лет
  3. Крадет с сервера приватный ключ
  4. Расшифровывает ВСЕ прошлые данные
  
Полный крах безопасности!
```

#### Что происходит в TLS 1.3:

```
Каждой сессии генерируются УНИКАЛЬНЫЕ одноразовые ключи
    ↓
Компрометация ключа сервера сегодня
    ↓
НЕ ПРИВОДИТ к расшифровке ВЧЕРАШНИХ данных
    ↓
Старые сессии остаются безопасными!
```

### Perfect Forward Secrecy гарантирует:

- Каждый сеанс использует свой уникальный ключ
- Компрометация текущего ключа не влияет на прошлые сессии
- Временное раскрытие ключа имеет ограниченный урон
